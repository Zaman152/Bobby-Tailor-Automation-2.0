## Artifacts this phase produces

New symbols created by this plan:
- **Function** `_capture_learning(run_folder, scope, item_pattern, original_value, corrected_value, extra_context) -> Optional[str]` in verify.py — shared helper: resolves project_key/project_type/sheet_type from the run's takeoff.json, calls `record_correction`, calls `distill()`, returns a plausibility warning string or None; never raises (capture failure logs + continues — the JSON write must not be rolled back by a learning error)
- **Response field** `"learning_warning"` (optional, additive) on verify/scale/measurements POST responses
- **Scope mapping** (documented in verify.py docstring): /verify → 'item' (or 'name'/'unit' when only those changed), /verify-batch → per-entry 'item', /scale → 'scale', /measurements → 'wall_height' or 'item' per measurement kind, missed additions → 'missed_item'
- No new endpoints, no URL changes.

<threat_model>
ASVS L1 — SQLite learning store + Flask endpoints:
- **Injection via learned strings (HIGH — mitigated upstream):** endpoint values flow to record_correction which is parameterized-only (21-04 guarantee). verify.py must NEVER construct SQL — acceptance grep asserts no sqlite3 import in verify.py.
- **Feedback amplification via typo'd corrections (MEDIUM):** RESEARCH Pitfall 8 — reuse QuantityVerifier plausibility rules at capture: an out-of-band corrected quantity persists (human is authoritative) but the response carries `learning_warning` so the UI can prompt re-check.
- **Auth/CSRF (HIGH — preserved):** endpoints keep `@login_required` + CSRF; no new unauthenticated surface. Path-traversal guards unchanged.
- **Data minimization:** context JSON stores page/viewport/provenance only — no session or user-agent data.
</threat_model>
